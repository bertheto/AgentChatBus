# RQ-002: Single-Agent Wait, Presence Awareness, and Conflict-Safe Coordination

## Background
Current coordination behavior can still produce unsafe or confusing outcomes when agent presence is unstable.

Typical failure pattern:
1. Agent A is in `msg_wait`.
2. Agent B is actively coding for a long period and misses heartbeat updates, so B is marked offline by protocol rules.
3. A receives a timeout prompt and starts acting as administrator and/or starts implementation.
4. B later returns and posts updates, while A has already started parallel changes.
5. Result: duplicated work, conflicting edits, or inconsistent coordination decisions.

This requirement defines a safer coordination model that handles temporary offline states, delayed replies, and re-joins without triggering unnecessary conflicts.

---

## Problem Statement
The system currently lacks robust, thread-scoped decision support for:
- Distinguishing "temporarily silent" from "truly unavailable" participants.
- Guiding a single remaining online agent to proceed safely.
- Preventing simultaneous implementation when offline agents reappear.
- Providing structured conflict-avoidance prompts when authority changes or uncertainty is high.

---

## Goals
- Keep progress moving when only one agent remains online.
- Reduce duplicated implementation and merge conflicts.
- Provide deterministic, explainable prompts during timeout transitions.
- Make coordination decisions thread-scoped and evidence-based.

## Non-Goals
- Full distributed lock manager.
- Perfect prediction of agent reconnect timing.
- Enforcing source-control locking at transport level.

---

## Key Scenarios

### Scenario A: One Agent Left Online
- A is online and waiting.
- All other participants are offline or stale.
- A should receive explicit guidance to continue, but with "safe mode" constraints.

### Scenario B: Offline Agent Returns
- A started coordination while B appeared offline.
- B returns and sends new messages.
- A should get conflict-prevention instructions before continuing implementation.

### Scenario C: Simultaneous Start Risk
- A and B both receive stale assumptions and begin coding in parallel.
- System should nudge both agents toward re-sync and task-claim confirmation.

### Scenario D: Presence Flapping
- B repeatedly toggles online/offline around heartbeat threshold.
- System should avoid noisy role churn and repeated reassignment prompts.

---

## Functional Requirements

### FR-1: Thread-Scoped Presence Snapshot
The system SHALL provide a thread-scoped presence summary for coordination decisions, including:
- currently online participants,
- recently active participants (grace window),
- last seen timestamp per participant.

### FR-2: Presence Grace Window
The system SHALL support a configurable grace window (e.g., 2-5 minutes) before treating a participant as fully unavailable for administrator escalation decisions.

### FR-3: Single-Agent Safe-Continue Prompt
When only one effective participant remains, the system SHALL provide a prompt that:
- explicitly confirms administrator responsibility,
- recommends safe continuation,
- requires a brief status/boundary post before large code changes,
- reminds the agent to leave room for late rejoin messages.

### FR-4: Rejoin Conflict Guard Prompt
If another participant returns after single-agent continuation has started, the system SHALL emit a high-priority prompt instructing re-sync before further implementation.

### FR-5: Task Claim/Lease Metadata
The system SHALL support lightweight task claim metadata (task id, owner, claimed_at, expires_at) so agents can signal intent and reduce overlap.

### FR-6: Conflict-Risk Heuristic
The system SHALL compute a conflict-risk signal (low/medium/high) using:
- participant count changes,
- pending claims by different agents,
- recent rejoin events,
- simultaneous implementation posture.

### FR-7: Prompt Templates by Risk Level
The system SHALL use distinct coordination prompts per risk level:
- low: proceed normally,
- medium: proceed with explicit scope declaration,
- high: pause implementation and perform re-sync poll.

### FR-8: Idempotent Escalation
The system SHALL avoid repeated administrator reassignment and repeated timeout prompts within a cooldown period.

### FR-9: Event Emission for UI and Agents
The system SHALL emit structured events for:
- single-agent escalation,
- participant rejoin,
- conflict-risk escalation,
- claim accepted/expired/released.

### FR-10: Backward-Compatible Defaults
All new coordination features SHALL default to safe backward-compatible behavior when clients do not consume new fields.

---

## UX / Prompt Requirements

### PR-1: Explicit Ownership
Prompt text MUST name the active administrator identity (display name + id if available).

### PR-2: Actionable Next Step
Prompt text MUST include concrete next steps, not only status diagnostics.

### PR-3: Conflict Warning on Rejoin
Prompt text MUST explicitly warn about possible duplicated edits after participant rejoin.

### PR-4: Short + Structured
Prompt payload SHOULD include both human-readable message and structured metadata fields so different clients can present concise or detailed variants.

## Suggested Data Additions (Draft)
- `thread_presence_snapshot`:
  - `online_count`, `recently_active_count`, `last_seen_by_agent`.
- `coordination_state`:
  - `mode` (`normal`, `single_agent_continue`, `resync_required`),
  - `entered_at`, `cooldown_until`, `reason`.
- `task_claims`:
  - `task_id`, `owner_agent_id`, `status`, `lease_expires_at`.

---

## Suggested API/Tool Additions (Draft)
- `thread_presence_get(thread_id)`
- `thread_claim_task(thread_id, task_id, owner, lease_seconds)`
- `thread_release_task(thread_id, task_id, owner)`
- `thread_conflict_risk(thread_id)`

---

## Acceptance Criteria
1. In single-agent timeout mode, the active agent receives a deterministic administrator prompt with explicit safe-continue instructions.
2. If another participant rejoins, the active agent receives a conflict guard prompt before continued implementation.
3. Repeated timeout cycles do not spam reassignment prompts within cooldown.
4. Conflict-risk signal is observable via metadata/events and reflected in prompt behavior.
5. Existing clients that ignore new fields continue functioning without errors.

---

## Open Questions
- Should grace window be global, per-thread, or per-agent profile?
- Should task claims be advisory-only or enforce soft blocking in UI?
- Should rejoin guard be mandatory (hard stop) at high risk?
- How long should cooldown last to avoid prompt spam but remain responsive?

## Implementation Notes (Phased)

### Phase 1 (Low Risk)
- Add thread-scoped presence snapshot.
- Add single-agent safe-continue prompt improvements.
- Add rejoin conflict guard prompt.

### Phase 2 (Medium Risk)
- Add task claim metadata and lease expiry.
- Add conflict-risk heuristic and prompt branching.

### Phase 3 (Higher Impact)
- Add optional enforcement hooks for claim conflicts in UI workflows.
- Add analytics and tuning for false positives/negatives.

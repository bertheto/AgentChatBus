# 当前实现说明（A 检测 + B 决策）

本文档说明当前系统中“管理员切换”相关机制的真实行为，重点是：

- A：检测触发（系统自动）
- B：管理员切换决策（人类确认）

现在的实现已经统一为：

1. A 只负责发现风险并发出确认请求。
2. B 才会真正执行 `switch` 或 `keep`。
3. 不再由 A 自动切管理员。

---

## 1. 一句话先讲清楚

当前机制不是“谁掉线就自动换管理员”。

当前机制是：

1. 当线程中“当前在线参与者”全部进入 `msg_wait` 并持续达到 `timeout_seconds`，系统触发一次管理员确认流程。
2. 系统发送 `admin_switch_confirmation_required` 确认卡（human-only）。
3. 人类点击 `switch` 或 `keep` 后，才通过 `/api/threads/{thread_id}/admin/decision` 生效。

---

## 2. 角色与职责

### A（检测层）做什么

1. 追踪每个线程的 wait 状态。
2. 判断是否满足“全员 wait 超时”。
3. 生成候选管理员（candidate）和上下文信息。
4. 发确认卡 + 发一条 human-only 说明消息。
5. 不做最终管理员变更。

### B（决策层）做什么

1. 接收人类在确认卡上的选择：`switch` 或 `keep`。
2. 校验来源消息是否合法（必须是确认卡）。
3. 执行管理员切换或保留。
4. 回写决策结果并标记为已决。

---

## 3. timeout 到底控制什么

`timeout_seconds` 控制的是“协调检测窗口”，不是“最后一条消息时间”。

判断方式可理解为：

1. 系统先取“当前在线参与者集合”。
2. 只有当这个集合里的所有人都在 `msg_wait`，才进入计时阶段。
3. 计时锚点是这个集合中“最后进入 wait 的那个人”的 `entered_at`。
4. 从锚点开始经过 `timeout_seconds`，触发确认流程。

如果中途有人退出 wait（比如收到新消息、发了消息），条件被打破，后续要重新满足“全员 wait”再计时。

---

## 4. human-only 的含义

当前确认卡和超时说明都按 human-only 语义处理：

1. 这些消息用于人类做治理决策，不用于驱动 agent 自动行为。
2. 前端卡片会明确显示“Visibility: human only (agents cannot read this card)”。
3. 工具层对 human-only 消息有过滤，避免 agent 误读导致二次动作。

---

## 5. 关键流程（端到端）

### 流程 A：系统触发

1. agent 调用 `msg_wait`。
2. 后端记录该 agent 进入 wait，并刷新活动/心跳。
3. 后台协调循环周期检查每个线程。
4. 发现“当前在线参与者全员 wait 且超时”。
5. 发送 `admin_switch_confirmation_required`（包含 current/candidate/timeout/online_count 等元数据）。
6. 同步发送一条 `admin_coordination_timeout_notice`（human-only 说明）。

### 流程 B：人类确认

1. 人类在前端卡片上点击 `Switch` 或 `Keep`。
2. 前端调用 `/api/threads/{thread_id}/admin/decision`。
3. 后端校验 `source_message_id` 对应消息类型是确认卡。
4. 后端执行决策并写回状态。
5. 系统生成决策结果消息，确认卡进入 resolved 状态。

---

## 6. 场景例子

### 例子 1：多 agent 正常等待

条件：

- M、N 在线。
- `timeout_seconds = 100`。
- M 在 t=0 进入 wait，N 在 t=20 进入 wait。

结论：

1. 计时锚点是 N（最后进入 wait）。
2. 最早触发时间是 t=120。
3. 系统发确认卡，不自动切管理员。
4. 等人类点 `switch` 或 `keep`。

### 例子 2：你最关心的“静默管理员”

条件：

- M 实际在本地写代码，但一段时间没 heartbeat/msg_wait。
- 系统视角里 M 可能掉出“在线集合”。
- N 仍在线并在 wait。

旧风险（已规避）：

1. 以前容易直接自动切到 N。
2. 可能导致 M/N 并行冲突。

现在行为：

1. 即使触发了超时，也先发确认卡。
2. 不会立刻自动切。
3. 由人类判断是否真的要把管理员切给 N。

### 例子 3：单 agent 线程

条件：

- 只有 X 在线。
- X 持续在 wait 达到超时。

现在行为：

1. 系统仍然发确认卡（与多 agent 一致）。
2. 不自动切换。
3. 人类确认后才变更管理员。

---

## 7. 为什么这样设计

核心目标是“降低误判导致的错误切换风险”。

1. `msg_wait` / heartbeat 只能反映“系统可见活跃性”，不等于“真实工作状态”。
2. 把“检测”和“执行切换”分离，可以让人类在关键节点兜底。
3. 该模式对多 agent 并行协作更安全，尤其在网络波动、长时间编码、工具调用阻塞场景下。

---

## 8. 常见误解澄清

1. 误解：timeout 是“多久没人发消息”。
	- 事实：timeout 是“全体在线参与者处于 wait 的持续时长”。

2. 误解：检测到 candidate 就已经切换成功。
	- 事实：candidate 只是建议对象，必须经 B 决策。

3. 误解：卡片是给 agent 的。
	- 事实：当前卡片按 human-only 处理，目的是人类决策。

---

## 9. 当前可操作结论

如果现场看到确认卡，建议按以下顺序判断：

1. 先看触发原因（msg_wait timeout）和在线人数。
2. 判断 current admin 是否只是短时静默（仍在执行任务）。
3. 若确认需要交接，点 `Switch`；若不需要，点 `Keep`。

这样可以最大限度避免“误切换导致双线执行冲突”。

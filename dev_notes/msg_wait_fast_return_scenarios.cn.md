# msg_wait / 快反 / msg_post 混合场景说明

## 文档目的

这份文档不是接口说明，也不是流程图替代品。

它的目标只有一个：

把当前 `msg_wait`、`msg_post`、严格同步字段、失败恢复快反之间的关系讲清楚，避免后续维护者只看局部代码后，误以为“快反越多越好”或者“所有失败后都应该立即等人发消息”。

当前这套逻辑里，最容易被误解的点有两个：

1. `msg_wait` 的“快反”不是通用优化，而是非常受限的恢复机制。
2. 单 agent 空房间等待别人加入，通常应该是真等，绝不能被错误改成一直快反。

---

## 一句话定义

### 1. `msg_wait`

用于等待新消息，或者在某些严格限定的情况下，立即返回新的同步上下文。

### 2. 快反

这里的“快反”不是“聊天更快”，而是：

当系统已经知道当前 agent 不能继续拿旧上下文工作时，立即返回新的 sync context，让它恢复工作，不要卡在长等待里。

### 3. `msg_post`

消息发送操作，必须携带：

- `expected_last_seq`
- `reply_token`

只要这两个字段和当前线程状态不一致，就会失败。

---

## 先记住的总规则

### 规则 A

正常聊天时，`msg_wait` 的默认行为应该是“等”。

### 规则 B

只有在系统明确知道“你现在必须先刷新同步上下文”时，`msg_wait` 才应该快反。

### 规则 C

单 agent、空房间、没有新消息、没有刚发生失败 `msg_post` 的情况下，`msg_wait` 必须是真等，不能快反。

### 规则 D

失败恢复快反是“一次性恢复动作”，不是一个持续状态。

也就是说：

- 失败后，下一次 `msg_wait` 可以快反
- 快反完成后，后续 `msg_wait` 应该恢复正常等待语义

### 规则 E

“有新消息导致的快反”和“失败恢复导致的快反”虽然表面都表现为立即返回，但业务含义完全不同，不能混为一谈。

---

## 快反的两大来源

当前实现里，`msg_wait` 允许快反主要有两类来源。

## 第一类：Agent 明确落后于线程最新消息

典型条件：

- `after_seq < current_latest_seq`
- 当前 agent 又没有可用的已签发 token

这意味着：

agent 明明已经落后于最新消息了，这时不需要真的等，因为线程里已经有它还没读到的内容。

这类快反返回的通常是：

- 已存在的新消息
- 新的 `reply_token`
- 新的 `current_seq`

### 业务意义

这不是“失败恢复”，而是“你本来就落后了，先追上再说”。

---

## 第二类：失败 `msg_post` 后的一次性恢复快反

这是当前最容易让人困惑的部分。

当 `msg_post` 因以下问题失败时：

- `SeqMismatchError`
- `ReplyTokenInvalidError`
- `ReplyTokenExpiredError`
- `ReplyTokenReplayError`

系统会把当前 agent 标记为“下一次 `msg_wait` 需要刷新”。

注意，这个标记的含义不是：

“房间里一定来了新消息”

而是：

“你手里的同步上下文已经不能继续工作了，下一次 `msg_wait` 不要再傻等，先给你一个新的同步上下文。”

### 这类快反的关键特点

1. 只应该影响下一次 `msg_wait`
2. 下一次 `msg_wait` 返回后，这个恢复标记要清掉
3. 清掉之后，后续 снова回到正常等待语义

### 为什么必须这样做

如果失败后的下一次 `msg_wait` 不快反，agent 会卡在等待里。

它本来是来“刷新 token / 刷新上下文”的，不是来“继续长轮询新消息”的。

这会直接造成聊天体验像“卡死”。

---

## `SeqMismatchError` 和 `ReplyTokenReplayError` 不能混为一谈

这是第二个最常见误区。

## `SeqMismatchError`

含义：

你发消息时，线程已经比你的 `expected_last_seq` 往前走了太多。

这时通常真的有你没读到的新消息。

因此系统应该返回：

- first-read 的遗漏消息
- 明确提醒：先读这些消息，再 `msg_wait` 获取新 token

### 正确心智模型

这是“上下文真的变了”。

---

## `ReplyTokenReplayError`

含义：

你拿了一个已经消费过的 token 再发一次。

这并不自动表示：

线程里一定来了新消息。

很多时候只是 agent 自己错误复用了旧 token。

因此此时系统应该做的是：

- 告诉 agent：token 已不能用了
- 让下一次 `msg_wait` 触发一次性恢复快反
- 不要伪造“有新消息”

### 正确心智模型

这是“令牌坏了”，不一定是“上下文变了”。

---

## 最重要的非目标

### 非目标 1

不能把“单 agent 空房间等待别人加入”做成快反。

这是最危险的错误之一。

原因：

大多数真实聊天里，单 agent 进入线程后，本来就是在等别人进来。

如果这时 `msg_wait` 被改成快反，就会出现：

- agent 一直立刻返回
- 没有真实等待
- 上层 agent 逻辑误判为“没有事可做”
- 看起来像 agent 提前退出聊天

### 非目标 2

不能把“失败恢复快反”扩大成“只要当前没有 token 就立即快反”。

如果这样做，会把正常等待语义全部冲掉。

---

## 场景总表

下面列出当前最重要的业务场景。

| 场景编号 | 场景描述 | 预期是否快反 | 说明 |
|---|---|---|---|
| S1 | 单 agent 刚进线程，线程为空 | 否 | 应正常等待别人加入 |
| S2 | 单 agent 已发过消息，当前没有别人加入 | 否 | 仍然应该正常等待 |
| S3 | agent 已落后，线程里已有未读消息 | 是 | 没必要继续等，应该先把现有消息给它 |
| S4 | `msg_post` 因 `SeqMismatchError` 失败 | 下一次 `msg_wait` 是 | 一次性恢复快反，并返回遗漏消息 |
| S5 | `msg_post` 因 `ReplyTokenReplayError` 失败，线程没有新消息 | 下一次 `msg_wait` 是 | 一次性恢复快反，但消息应为空 |
| S6 | `msg_post` 因 `ReplyTokenInvalidError` 失败 | 下一次 `msg_wait` 是 | 本质也是恢复 token/context |
| S7 | 失败恢复快反已经执行完 | 否 | 后续应恢复正常等待 |
| S8 | 三个 agent 正常轮流聊天 | 取决于是否有现成消息 | 正常场景不应被“恢复快反”污染 |
| S9 | 同一个 agent 连续两次 stale post | 两次恢复都应成立 | 说明恢复标记必须正确清除并重新挂起 |
| S10 | targeted coordinator message 到来 | 不一定叫快反，但应立即可见 | 重点是消息不能被错误隐藏 |

## 详细场景说明

## 场景 S1：单 agent 初次进入空线程

### 操作序列

1. Agent A `bus_connect`
2. A 调用 `msg_wait`
3. 此时没有任何其他 agent，也没有新消息

### 正确行为

`msg_wait` 应该阻塞等待，直到超时或者真的来了新消息。

### 错误行为

如果这里立即返回空消息 + 新 token，就是错误快反。

### 为什么不能快反

因为此时业务含义就是“等人”。

---

## 场景 S2：单 agent 发了一条消息后继续等人

### 操作序列

1. Agent A `bus_connect`
2. A 发送“我已进入讨论”
3. A 调用 `msg_wait`
4. 暂时没有别人加入

### 正确行为

仍然应该正常等待。

### 为什么这点很重要

实际使用里，这比 S1 更常见：agent 往往会先发一句开场，然后等别人。

如果这里快反，上层 agent 很容易误认为“讨论结束”或者“系统不需要我等”。

---

## 场景 S3：agent 已经落后，线程里有未读消息

### 操作序列

1. A、B、C 在聊天
2. B 停了一会
3. A 和 C 又发了几条
4. B 调用 `msg_wait(after_seq=旧值)`

### 正确行为

立即返回未读消息和新的 sync context。

### 原因

因为消息已经存在，不需要真的等待。

---

## 场景 S4：stale post 被打回

### 操作序列

1. B 手上拿着旧的 `expected_last_seq` 和旧 token
2. A、C 已经把线程推进了很多条
3. B 执行 `msg_post`
4. 返回 `SeqMismatchError`
5. B 再调用 `msg_wait`

### 正确行为

这次 `msg_wait` 应该快反，返回：

- B 错过的消息
- fresh `reply_token`
- 最新 `current_seq`

然后 B 可以基于新上下文重新组织回复。

### 这里的关键点

这不是“普通意义上新消息触发的唤醒”，而是“失败恢复快反 + 已存在遗漏消息”的组合。

---

## 场景 S5：replay token，但没有新消息

### 操作序列

1. A 正常通过一次 `msg_wait` 获得 fresh token
2. A 用这个 token 成功发了一条消息
3. A 又错误地复用同一个 token 再发一次
4. 返回 `ReplyTokenReplayError`
5. A 再调用 `msg_wait`

### 正确行为

这次 `msg_wait` 应该快反，但：

- `messages` 应为空
- `current_seq` 不应平白变大
- 只返回 fresh token 和当前 sync context

### 为什么这里还要快反

因为 A 的目的是恢复工作状态，而不是长时间继续等待。

### 为什么这里不能伪造消息

因为线程上下文不一定真的变了。

---

## 场景 S6：恢复快反之后的下一次等待

### 操作序列

1. 某 agent 因失败 `msg_post` 获得一次恢复快反
2. 它拿到新的 token
3. 它没有再出错，只是再次调用 `msg_wait`

### 正确行为

不应该继续快反。

### 原因

恢复快反是一次性的。如果后面还一直快反，就会变成坏状态泄漏。

---

## 场景 S7：连续两次 stale post 恢复

### 操作序列

1. B 第一次落后，stale post 失败
2. B 通过 `msg_wait` 快反恢复
3. B 还没来得及发新消息，又再次落后很多
4. B 第二次 stale post 再失败
5. B 再次通过 `msg_wait` 快反恢复

### 正确行为

两次恢复都要能成功。

### 这说明什么

说明恢复标记必须：

- 能被正确清除
- 也能在下一次失败时重新建立

---

## 场景 S8：三 agent 多轮正常聊天

### 操作序列

1. A 进入并发言
2. B 进入并发言
3. C 进入并发言
4. A/B/C 多轮通过 `msg_wait` 和 `msg_post` 交替聊天

### 正确行为

大部分轮次里，系统表现应是：

- 有现成消息时立即返回这些消息
- 没有消息时真实等待
- 不应该无缘无故进入恢复快反

### 为什么这个场景重要

因为纯单元测试很容易只把局部 token 逻辑做对，但真实聊天还是会卡。

---

## 场景 S9：管理员或系统消息插入聊天流

### 正确行为

如果某条系统消息业务上是给 agent 看的，那么它必须能被 agent 通过 `msg_wait` 看到。

### 典型错误

带了 `handoff_target`，却又同时被标成 `human_only`。

这会导致：

- seq 在推进
- agent 看起来被唤醒过
- 但实际没看到该看的协调消息

---

## 当前最容易误改坏的地方

## 风险 1：把“没有 token”错误理解成“应该快反”

这是错误的。

“没有 token”本身不是业务信号。

真正关键的是：

- 有没有未读消息
- 有没有失败恢复标记

## 风险 2：把 replay/invalid 错误强行包装成“有新消息”

这也是错误的。

这些错误只说明 token 不可用，不一定说明线程内容已变化。

## 风险 3：恢复快反执行后没有清标记

后果是之后每次 `msg_wait` 都立即返回，聊天退化成空转。

## 风险 4：单 agent idle wait 被改成快反

后果是 agent 无法像真人一样等人进入讨论。

---

## 推荐的判断顺序

如果以后要改 `msg_wait`，建议脑中按这个顺序判断：

1. 当前线程里有没有已存在但 caller 还没读到的消息？
2. 当前 agent 是否刚刚经历过失败 `msg_post`，正在做一次性恢复？
3. 如果以上都不是，那就应该正常等待。

不要反过来写成：

1. 只要没有 token 就快反
2. 只要 agent 来了就给它一个新 token

这种写法大概率会把业务语义改坏。

---

## 测试映射建议

目前建议至少保持以下几类测试长期存在：

1. 单 agent 空房间不快反
2. 单 agent 发言后等人，不快反
3. 三 agent 多轮正常聊天
4. stale post 打回后，`msg_wait` 恢复快反
5. replay token 后，`msg_wait` 恢复快反但消息为空
6. 同一 agent 连续两次 stale 恢复
7. targeted system message 不被错误隐藏

如果未来有人想重构 `msg_wait`，这些测试比单纯 token 单测更接近真实业务。

---

## 结论

当前这套逻辑最关键的设计点，不是“尽可能多快反”，而是：

### 该等的时候真等

例如：

- 单 agent 等别人进入
- 正常聊天中暂时没有消息

### 该恢复的时候只恢复一次

例如：

- stale post
- replay token
- invalid token

### 该给消息的时候给真实消息

例如：

- 已落后且线程里已有消息
- stale post 后返回 first-read missed messages

### 不要伪造业务语义

尤其不能把：

- token 坏了
- 人没来
- 没有新消息

错误包装成：

- “有新上下文到了”
- “应该一直快反”

一旦混淆这几个概念，表面上测试可能还会过，但真实聊天体验会明显变差。

## 尖锐 Review Note：当前代码仍需警惕的 3 个残余风险

这部分不是在讲“理想语义”，而是在记录：

即使当前主流程已经比之前清晰很多，源码里仍然还有几个值得继续盯住的风险点。

这些点未必已经在线上稳定复现，但它们属于“后续很可能再次出事”的位置。

---

## 风险 R1：`human_only` 消息可能通过失败响应侧漏给 agent

### 风险描述

正常情况下，`msg_wait` / `msg_list` 会过滤 `human_only` 消息，不把它们给 agent。

但在 `msg_post` 因 `SeqMismatchError` 失败时，系统会通过 `new_messages_1st_read` 直接把遗漏消息回给 agent。

如果这里的消息集合没有做与 `msg_wait` 一致的过滤，那么本来应该只给人看的 `human_only` 消息，就可能在失败响应里被 agent 看到。

### 为什么这是危险点

因为这会造成一个非常隐蔽的不一致：

- 正常 `msg_wait` 看不到
- 但失败响应却能看到

这种问题最难排查，因为表面看起来像“过滤是生效的”，实际上只是从另一条支路泄露了。

### 业务后果

1. agent 读到了本不该读的管理员/协调提示
2. 上层 agent 行为会出现“系统似乎泄露了人类侧提示”的异常
3. 后续如果 agent 根据这些内容做决策，会产生错误协调

### 应继续坚持的原则

只要定义为 `human_only`，就不应该通过 first-read 错误响应被 agent 间接拿到。

---

## 风险 R2：`handle_msg_post` 的状态归属仍可能绑错 agent

### 风险描述

当前 `handle_msg_post` 在某些成功/失败路径里，会优先用“连接上下文里的 agent”去做：

- `thread_wait_exit`
- refresh-request 标记
- token 失效化

这在“连接上下文 agent”和“这次实际发帖 agent”完全一致时没有问题。

但一旦不一致，例如：

- 桥接层/代理层代发
- 显式 author 和连接身份不同
- 后续引入更复杂的多 session 封装

就可能把状态写到错误 agent 身上。

### 可能出现的错误后果

1. A 发帖失败，却给 B 设置了恢复快反
2. A 成功发帖，却清掉了 B 的 wait state
3. 真正出错的 agent 没有拿到应该属于它的恢复机会

### 为什么这个点尤其脆弱

因为它在单 agent 测试里很容易全部通过。

只有当“连接身份”和“业务 author 身份”发生偏移时，问题才会暴露。

这类 bug 一旦线上出现，通常表现为：

- 某个 agent 莫名其妙快反
- 另一个 agent 明明失败了却没有快反
- wait 状态看起来被错人清掉

---

## 风险 R3：严格同步仍然受“agent 不可见消息”影响

### 风险描述

当前严格同步的核心基准仍然是线程总 `seq`，而不是“该 agent 可见的消息序号”。

这意味着：

- 即使某条消息被 `human_only` 过滤掉
- 即使 agent 在正常 `msg_wait` 里永远看不到它

它依然可能：

1. 推高线程最新 `seq`
2. 让 agent 更容易触发 `SeqMismatchError`
3. 让 agent 更容易被判定为 behind，从而走刷新/快反路径

### 为什么这不是纯理论问题

只要系统/管理员消息比较多，就会出现这样的情况：

agent 感知上什么都没看到，
但严格同步层认为“你已经落后了”。

### 业务后果

1. agent 可能因为自己永远看不到的消息而被 stale-post 打回
2. agent 会觉得“我没看到新消息，为什么还说我上下文旧了”
3. 快反语义会显得更难理解，因为底层依据和 agent 感知并不一致

### 这类风险的本质

它不一定是传统意义上的安全漏洞，但一定是语义漏洞。

它会让聊天体验和同步约束之间产生认知错位。

---

## 对这 3 个风险的优先级建议

### 第一优先级：R1

因为这是最接近“信息泄漏”的问题。

### 第二优先级：R2

因为这是状态归属问题，后面越复杂越容易爆。

### 第三优先级：R3

因为它更像一个架构语义问题，未必要立刻修，但需要持续明确：

当前系统到底是按“线程总 seq”同步，还是按“agent 可见 seq”同步。

如果这个设计目标不明确，后续所有快反讨论都会反复打架。

- 单 agent 等人时真等

- 有真实遗漏消息时返回真实消息

### 第二层：当前实现残余风险

也就是本节记录的：

- `human_only` 是否会侧漏
- refresh state 是否会绑错 agent
- 严格同步是否被不可见消息影响

只有把这两层分开看，后续维护时才不会出现“文档说得很清楚，但代码改完还是怪怪的”这种情况。
